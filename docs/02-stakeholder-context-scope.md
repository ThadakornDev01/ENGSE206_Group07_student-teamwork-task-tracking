## 1. Problem Frame (revised)

### 1.1 สถานการณ์ปัจจุบัน
นักศึกษามักมีภาระงานกลุ่มหลายรายวิชา แต่ยังคงใช้การสื่อสารและสั่งงานผ่านแอปพลิเคชันแชตเป็นหลัก ทำให้ข้อมูลและไฟล์งานกระจัดกระจาย ไม่มีความชัดเจนว่าสมาชิกคนใดรับผิดชอบงานส่วนไหน และบางคนอาจไม่ทราบกำหนดส่งที่แน่ชัด นำไปสู่ความยากลำบากในการประเมินผลการทำงานในช่วงท้าย

### 1.2 ใครได้รับผลกระทบ
- **สมาชิกกลุ่ม / ผู้ทำงาน:** กังวลเรื่องงานกระจัดกระจาย ไม่ทราบกำหนดส่งที่แน่ชัด และกลัวถูกประเมินการมีส่วนร่วมอย่างไม่เป็นธรรม
- **หัวหน้ากลุ่ม:** ตามงานยากหากสมาชิกไม่ส่งงาน หรือเกิดความสับสนเมื่อมีการเปลี่ยนบทบาทการทำงานกลางคัน
- **อาจารย์ผู้สอน:** ขาดหลักฐานและข้อมูลสนับสนุนในการประเมินคะแนนการทำงานกลุ่มอย่างเป็นธรรม
- **เพื่อน Reviewer:** ขาดหลักฐานหรือข้อมูลที่ชัดเจนในการประเมินการมีส่วนร่วม (Peer evaluation) ของเพื่อนร่วมทีม

### 1.3 Problem Statement
> นักศึกษาไม่สามารถจัดการภาระงานกลุ่มได้อย่างมีประสิทธิภาพเพราะการสั่งงานและส่งไฟล์ผ่านแชตทำให้ข้อมูลกระจัดกระจาย ส่งผลให้การแบ่งบทบาทไม่ชัดเจน และยากต่อการตรวจสอบหลักฐานการทำงานเมื่อต้องประเมินผล

### 1.4 ผลลัพธ์ที่ต้องการ (โดยยังไม่กำหนด solution)
- สมาชิกทีมมองเห็นภาระงาน บทบาท ความคืบหน้า และกำหนดส่งอย่างชัดเจน
- หัวหน้ากลุ่มสามารถติดตามการทำงานของสมาชิกได้อย่างเป็นระบบ
- มีการเก็บหลักฐานการมีส่วนร่วมของสมาชิกอย่างยุติธรรมเพื่อการประเมินตนเองก่อนส่งงาน
- อาจารย์ผู้สอนสามารถมองเห็นภาพรวมการทำงานเพื่อใช้ประเมินผลได้แบบพอดี

### 1.5 สิ่งที่ทีมยังต้องเรียนรู้
- งานในส่วนใดบ้างที่อาจารย์ควรเห็น และต้องการความละเอียดระดับใด
- หากเปลี่ยนเจ้าของงานกลางคัน ต้องเก็บประวัติเดิมไว้หรือไม่
- หลักฐานการมีส่วนร่วมรูปแบบใด (ไฟล์, ลิงก์, ข้อความ) ที่ถือว่าน่าเชื่อถือที่สุด
- ช่องทางการรายงานอุปสรรคการทำงานที่เหมาะสมสำหรับผู้ใช้

### 2.1 Stakeholder Profiles

#### Stakeholder: หัวหน้ากลุ่ม
- **Goal / need:** มอบหมายงานและดูว่าใครทำถึงไหนแล้ว เพื่อแก้ปัญหาความล่าช้าได้ทัน
- **Pain point:** ข้อมูลในแชตทำให้แบ่งงานยาก ตามงานยากเวลาคนไม่ส่ง และสับสนเมื่อมีการเปลี่ยนคนทำ
- **Concern / risk:** งานเสร็จไม่ทันกำหนด สมาชิกไม่ให้ความร่วมมือ
- **Influence / decision power:** High — มีอำนาจมอบหมายงานและเปลี่ยนตัวผู้รับผิดชอบ
- **Open questions for Week 3:** หากเปลี่ยนคนทำกลางคัน ระบบต้องเก็บประวัติคนเก่าไว้แบบไหน?

#### Stakeholder: อาจารย์ผู้สอน
- **Goal / need:** ดูหลักฐานว่าใครมีส่วนร่วมกับโปรเจกต์อย่างไรบ้างเพื่อประกอบการให้คะแนน
- **Pain point:** มองไม่เห็นภาพรวมการทำงานกลุ่ม ขาดหลักฐานที่น่าเชื่อถือตอนให้คะแนน
- **Concern / risk:** การให้คะแนนที่ไม่เป็นธรรม หรือไม่สามารถตรวจสอบความคืบหน้าก่อนวันพรีเซนต์ได้
- **Influence / decision power:** High — เป็นผู้ประเมินผลชี้ขาด
- **Open questions for Week 3:** อาจารย์ต้องการเห็นภาพรวมแบบใด? (กราฟ, สรุปรายบุคคล, หรือรายการงานค้าง?)

#### Stakeholder: สมาชิกกลุ่ม
- **Goal / need:** เห็นหน้าที่และกำหนดส่งงานของตัวเองชัดเจน เพื่อป้องกันการหลงลืม
- **Pain point:** งานกระจัดกระจาย ไม่เห็นกำหนดส่ง เสี่ยงถูกประเมินว่าไม่ช่วยเพื่อนทำงาน
- **Concern / risk:** กังวลว่าจะถูกประเมินอย่างไม่เป็นธรรมหากไม่สามารถพิสูจน์การมีส่วนร่วมได้
- **Influence / decision power:** Low — เป็นผู้ปฏิบัติตามแผน
- **Open questions for Week 3:** สะดวกอัปเดตงานบ่อยแค่ไหน? อุปสรรคในการแนบหลักฐานคืออะไร?

## 3. System Context

อธิบายขอบเขตระบบ ความสัมพันธ์กับผู้ใช้ และข้อมูลที่เข้าออก เพื่อการทำงานกันเป็นระบบและสามารถประเมินตัวเองได้ก่อนส่งงาน

### 3.1 System Boundary
ขอบเขตของระบบครอบคลุมเฉพาะการแบ่งภาระงาน การอัปเดตความคืบหน้า การแนบหลักฐานการทำงาน และการเก็บข้อมูลสำหรับการประเมิน Peer Evaluation ภายในขอบเขต 1 รายวิชา/1 โครงการ โดย **ไม่รวม** ระบบแชตพูดคุย และไม่เชื่อมต่อกับระบบฐานข้อมูลคะแนนของมหาวิทยาลัย

### 3.2 Key Data Flows

| From → To | Data / Request | Why it matters |
|---|---|---|
| หัวหน้ากลุ่ม → ระบบ | กำหนดการแบ่งงาน, ผู้รับผิดชอบ, วันกำหนดส่ง | เป็นข้อมูลตั้งต้นให้สมาชิกทุกคนรู้หน้าที่ |
| ระบบ → หัวหน้ากลุ่ม | สถานะความคืบหน้า, แจ้งเตือนงานล่าช้า | ช่วยให้หัวหน้าตามงานและแก้ปัญหาได้ตรงจุด |
| สมาชิกกลุ่ม → ระบบ | อัปเดตความคืบหน้า, ไฟล์/ลิงก์หลักฐาน | เป็นข้อมูลสำคัญสำหรับนำไปใช้ประเมินผล |
| ระบบ → สมาชิกกลุ่ม | รายการงานที่ต้องทำ, แจ้งเตือนกำหนดส่ง | ลดความผิดพลาดและงานตกหล่น |
| อาจารย์ผู้สอน → ระบบ | คำขอเรียกดูภาพรวมโปรเจกต์ | เพื่อตรวจสอบความคืบหน้าก่อนการนำเสนอ |
| ระบบ → อาจารย์ผู้สอน | Dashboard ภาพรวม, หลักฐานการมีส่วนร่วม | เพิ่มความน่าเชื่อถือในการให้คะแนน |
| เพื่อน Reviewer ↔ ระบบ | ขอเรียกดู Worklog / รับข้อมูล Worklog | ใช้เป็นข้อมูลประกอบการทำ Peer Evaluation |

## 4. Scope Statement

### In Scope
1. การจัดการภาระงาน (การแสดงบทบาท ความคืบหน้า วันกำหนดส่ง)
2. การแนบหลักฐานการทำงาน (รองรับไฟล์, ลิงก์, หรือข้อความ)
3. ระบบรองรับกระบวนการและข้อตกลง Peer Evaluation ภายในกลุ่ม
4. รองรับการทำงานในระดับ 1 รายวิชา / 1 โครงการ

### Out of Scope
1. ไม่รวมระบบ Chat หรือ Video call ภายในตัว
2. ไม่เชื่อมต่อกับระบบให้คะแนนหรือระบบบริหารการศึกษาของมหาวิทยาลัย
3. ไม่ทำระบบ Timesheet หรือระบบบริหารโครงการระดับองค์กร

### Constraints
| ID | Constraint / Rule | Source | Impact |
|---|---|---|---|
| C-01 | บทบาทการทำงานของสมาชิกสามารถสลับ/เปลี่ยนแปลงได้ตามความเหมาะสม | Case Card 07 | ระบบต้องออกแบบให้ยืดหยุ่นต่อการเปลี่ยนเจ้าของงาน |
| C-02 | ระบบเป็นเพียงเครื่องมือติดตามผล ไม่ได้เชื่อมต่อกับการตัดเกรดของอาจารย์โดยตรง | Case Card 07 | ไม่ต้องออกแบบ API เชื่อมกับระบบของมหาลัย |
| C-03 | การเปลี่ยนสถานะงานเป็น "เสร็จสิ้น" จะต้องมีการแนบหลักฐาน (ไฟล์/ลิงก์/ข้อความ) เสมอ | Team Assumption | ต้องบังคับ Validate ข้อมูลก่อนกดส่งงาน |
| R-01 | ผู้ใช้จะต้องมีอุปกรณ์สมาร์ตโฟน/คอมพิวเตอร์ที่เชื่อมต่อเครือข่ายอินเทอร์เน็ตได้ | Team Assumption | ระบบควรออกแบบเป็น Web-based หรือรองรับ Mobile |

## 5. Privacy, Ethics, Security and Responsible AI

| ประเด็น | การตัดสินใจ/ข้อควรระวังใน Case นี้ |
|---|---|
| Data minimization | เก็บเฉพาะข้อมูลที่เกี่ยวกับการทำงานและหลักฐาน ไม่เก็บข้อมูลส่วนตัวของนักศึกษาที่ไม่จำเป็น |
| Role-based access | เพื่อนกลุ่มอื่นในรายวิชาเดียวกันไม่สามารถเข้าถึงข้อมูลและหลักฐานของกลุ่มเราได้ |
| Transparency | ทุกการเปลี่ยนแปลง (เช่น การสลับงาน, การลบหลักฐาน) ควรมีการบันทึกไว้เพื่อให้ตรวจสอบย้อนหลังได้ |
| Responsible AI | หากใช้ AI ช่วยร่างโครงสร้างงาน ทีมต้องทวนสอบข้อมูลเสมอ และไม่ใช้ AI ในการสร้างหลักฐานเท็จ (Fake Evidence) |
